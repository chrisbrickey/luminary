## Issue Analysis
Description of top failure modes (what failed and why), including representative examples. 

**Issue 1: Retrieval relevance**

Failing to retrieve chunks that the golden dataset identifies as the most relevant is the primary failure mode (average score 0.38, well below 0.80 threshold). 

On average, fewer than 2 of the 5 retrieved chunks match expectations, with the majority being completely irrelevant. 
The problem is more pronounced with french queries (retrieval_relevance average in french 0.281 vs 0.481 in english).

The irrelevant chunks are thematically adjacent passages from the same source. 
An example are chunks about Locke's epistemology or the Quaker etiquette surfacing for "religious tolerance" 
instead of the specific passages that the golden dataset expects. 
This suggests that the embeddings capture broad topic proximity but lack the precision to distinguish among closely related passages.  


**Issue 2: Citation retrieval language**

The LLM produces citations in a different language that what the metric expects, 
which causes the `citation_to_retrieval_consistency` metric to fail (false positive).                                                                                                                      
   
In `science_moral_progress_en`, the model generated citations as "Letters Philosophiques 1734" (anglicized) 
instead of the correct French title "Lettres Philosophiques 1734" (original french). 
The consistency checker uses exact string matching against retrieved source titles. 
So all three inline citations scored as hallucinated, yielding a consistency score of 0.0 for that example. 

The issue is only present in english examples. All three french examples have consistency = 1.0 (no false positives) 
because the model renders the citations in the original french when it is prompted in French.

This is a translation issue, not a retrieval issue. The LLM translated the source titles to match the user's language.

**Issue 3: Multipage citations**
  
Aggregated multi-page citations are not matched by the consistency checker, which causes the metric to fail (false positive).                                                                                                                                             
   
In `religious_tolerance_en`, the model appended a summary citation at the end of the response: `[source: Lettres Philosophiques 1734, pages 1, 4, and 5]`. 
The checker does not recognize a match between this multi-page format and any of the individually, retrieved source title (single-page strings like `Lettres Philosophiques 1734, page 4`).
So this reduces the score for this metric even though all three pages that the model returned are correct.

**Issue 4: Language content volatility**

This issue was discovered when running multiple eval runs with different value of `DEFAULT_K`.

The metric `language_content_compliance` bounced non-monotonically across the four eval runs (1.0 → 0.833 → 1.0 → 0.833), 
as number of chunks retrieved per query (k) was tuned. But retrieval chunk count should not influence the generation-side 
language of the response. With only three English examples in the golden dataset, a single response drifting into French swings the metric by 0.33.
This metric could be stabilized by adding more English examples to the golden dataset.

**Issue 5: Vague summary citation**

This issue appeared only one time during experimentation, after running the eval against the updated golden dataset.

The example `science_moral_progress_en` emitted the following summary citation: `[source: Lettres Philosophiques 1734, various pages]`. 
The logic added to break down a multi-line citation (in response to Issue 3) does not cover this type of vague reference 
because there is nothing to parse from "various pages". When this issue occurred, the score for that example dropped to 0.80 on `citation_to_retrieval_consistency`.


## Changes Made
Listing of changes made (e.g., code, prompts, configs, datasets), including rationales.

### Issue 1: Retrieval Relevance

Increase `DEFAULT_K` from 5 to between 8 and 10 (more chunks per retrieval); k=9 seems to be the best choice after experimentation
- Rationales: Retrieving more chunks per query increases the likelihood that the most relevant chunks will be retrieved.
- Pros: low effort, targeted change
- Cons: trades a bit of precision for improved recall (F1 score); Golden dataset was designed for k=5 so we may need to update the golden dataset in order to better reflect expectations.

Experiment progression: Tuned default chunks per query (`DEFAULT_K`) 5 → 8 → 10 → 9

|                   | K=5   | K=8   | K=10  | K=9   |
|-------------------|-------|-------|-------|-------|
| Overall pass rate | 0.733 | 0.767 | 0.800 | 0.767 |
| Overall average   | 0.835 | 0.856 | 0.891 | 0.864 |
| Retrieval F1      | 0.381 | 0.447 | 0.456 | 0.489 |
| Avg recall        | 0.400 | 0.608 | 0.725 | 0.725 |
| Avg precision     | 0.367 | 0.354 | 0.333 | 0.370 |

Conclusion: K=9 is the sweet spot. Beyond K=9, additional chunks add irrelevance faster than they recover missing context.
- The K=5 → K=8 step produced the biggest gain by recovering recall (0.400 → 0.608) at almost no precision cost. 
- K=10 nudged recall higher (0.725) but precision continued to erode (0.333) and F1 nearly plateaued (+0.009 over K=8).
- At K=9, recall is already saturated (identical to K=10) while one fewer irrelevant chunk per query lifts precision back to 0.370 and yields the highest retrieval F1 in the experiment (0.489). 

Relevant eval runs:
- `2026-04-30T18-33-59.json` (k=5)
- `2026-05-27T20-02-17.json` (k=8)
- `2026-05-27T20-37-24.json` (k=10)
- `2026-05-27T20-44-18.json` (k=9)

_NB: The dips/gains on `language_content_compliance` has a different root cause and is detailed in other sections._

### Issue 2: Citation Language 

Remove anglicized source examples in the persona prompt and add a defensive rule such as 
`Source titles in citations MUST be reproduced verbatim from the passages, in their original language. Never translate, anglicize, or alter source titles.`

Rationale: The model is sometimes (transiently) translating "Lettres" to "Letters" when answering in English according to the localization functionality. 
That's not ideal for users because source titles are proper nouns that should be displayed verbatim. The metric surfaces this issue when it occurs. 

Pros: low effort, targeted change
Cons: updating the prompt does not guarantee that the LLM will adhere to the rules and examples consistently

Post-change observation: The prompt rule produced no measurable improvement on the targeted metric
because the metric was already saturated before this change landed.

|                                   | k=5 (original) | k=8     | k=10    | k=9 (pre-change) | k=9 (post-change) |
|-----------------------------------|----------------|---------|---------|------------------|-------------------|
| Commit                            | fde3ed0        | 8389b83 | e5d6d12 | b1195bf          | aadb526           |
| citation_to_retrieval_consistency | 0.79           | 1.00    | 1.00    | 1.00             | 1.00              |
| Overall pass rate                 | 0.733          | 0.767   | 0.800   | 0.767            | 0.800             |
| Overall average                   | 0.835          | 0.856   | 0.891   | 0.864            | 0.898             |

Findings:
- The anglicization of sources (e.g. `"Letters Philosophiques"`) only appeared in the original run (k=5). The anglicization of sources did not occur in any of the following runs. After the first experiment for tuning k (k=8), all EN responses included sources in verbatim french: `[source: Lettres Philosophiques 1734, page N]` and `citation_to_retrieval_consistency` rose to 1.0. That was well before the new prompt rule was added. 
- The overall_average improvement between the pre-change (0.864) and post-change (0.898) runs is entirely driven by `language_content_compliance` flipping 0.83 → 1.0, which Issue 4 above identifies as volatile due to N=3 EN examples. This is not a signal of citation behavior.
- My best guess is that `Issue 2` was resolved (at least temporarily) due to the model's nondeterminism on the relatively small sample (n=3) of english examples.

Conclusion: 
Defensively, I made the changes to the personal prompt in order to codifify the desired behavior. This hedges against regression on future sources, model swaps, and expansions of the english dataset. 
However, this change cannot be credited with an eval-measurable improvement at this time. With only 3 non-french examples, the metric saturates at 1.0 easily. Expansion of the dataset (recommended for Issue 4 also) would also help to prevent regression.

### Issue 3: Multipage Citations

Expand a multi-page citation into its single-page equivalents before matching. Then run the existing substring matcher.

Rationale: 
For now, the presence of a citation summary is a user feature that I do not want to remove just to make the evaluations easier.
So in order to ensure the summarized multi-page citations are correct, we must evaluate them as individual requirements.
  
Pro: covers both inline and summary citations (does not ignore the summary citations when running eval)
Con: doesn't explicitly distinguish between inline citations and summarized citations

Post-change observation: The expansion logic produced no measurable improvement on the targeted metric
because the metric was already saturated before this change landed.

|                                   | k=5 (original) | k=8     | k=10    | k=9 (pre-change) | k=9 (post-change) |
|-----------------------------------|----------------|---------|---------|------------------|-------------------|
| Commit                            | fde3ed0        | 8389b83 | e5d6d12 | 5e12c6b          | e811ef9           |
| citation_to_retrieval_consistency | 0.79           | 1.00    | 1.00    | 1.00             | 1.00              |

| Overall average                   | 0.835          | 0.856   | 0.891   | 0.898            | 0.898             |

Findings:
- The summary multi-page citation (e.g. `[source: Lettres Philosophiques 1734, pages 1, 4, and 5]`) only appeared in the original run (k=5) on `religious_tolerance_en`. It did not reappear in any subsequent run. From k=8 onward, every example emitted only inline single-page citations (e.g. `[source: Lettres Philosophiques 1734, page N]`), which the existing substring matcher already handled.
- All aggregate metrics in the post-change run (e811ef9) are identical to the pre-change run (5e12c6b), including `retrieval_relevance`. With no summary citations in the response text, the new expansion logic has nothing to act on. So the metric is unchanged.
- My best guess is that `Issue 3`, like `Issue 2`, was resolved (at least temporarily) due to the model's nondeterminism on the relatively small number of golden examples.

Conclusion:
Defensively, I added the expansion logic to the consistency metric in order to evaluate summary citations fairly if they reappear.
This change cannot be credited with an eval-measurable improvement at this time. 

### Update Golden Dataset

After implementing the above changes (including tuning of `DEFAULT_K` from 5 to 9), 
I regenerated the golden dataset via the automated functionality and guidance that I previously developed, which harnesses LLM-based independent drafting and evaluation. 
I expanded the configuration to cover two more topics supported by the corpus, which resulted in four additional examples: two english and two french.
Then I ran the eval harness using the updated golden dataset. Below are results and analysis.

Eval re-run on commit `194c7db` against `persona_voltaire_v3.0_2026-05-30` (artifact `2026-05-30T12-05-01.json`).

| Metric                            | k=9 v2.0 (e811ef9, n=6) | k=9 v3.0 (194c7db, n=10) |
|-----------------------------------|-------------------------|--------------------------|
| Overall pass rate                 | 0.800                   | 0.800                    |
| Overall average                   | 0.898                   | 0.892                    |
| citation_accuracy                 | 1.000                   | 1.000                    |
| citation_to_retrieval_consistency | 1.000                   | 0.980                    |
| retrieval_relevance (F1)          | 0.489                   | 0.482                    |
| Avg recall / precision            | 0.725 / 0.370           | 0.643 / 0.389            |
| EN F1 / FR F1                     | -                       | 0.541 / 0.424            |
| language_metadata_compliance      | 1.000                   | 1.000                    |
| language_content_compliance       | 1.000                   | 1.000                    |

Findings:
- **Stability at k=9 was validated.** Even though I nearly doubled the number of examples (from 6 to 10) and expanded the topics covered, 
  the aggregate scores barely changed: pass rate identical at 0.800, overall average 0.898 → 0.892, F1 0.489 → 0.482.
  This is evidence that the improved results in the previous eval run were not the result of a small sample of topics tested.
  Said another way, this latest eval run supports the conclusions made when tuning k in previous sections (e.g., "k=9 is the sweet spot," "F1 plateaus around 0.49").
- **Retrieval is still the dominant gap.** The v3.0 dataset expects more chunks per query (4–6 vs ~3 in v2.0), which should be more
  realistic given the updated value of `DEFAULT_K`. But recall fell (−0.08) because the more demanding dataset makes it more
  difficult to ensure that expectations are met (that all critical chunks are retrieved). At fixed k=9, recall dropped 0.725 → 0.643 
  while precision rose 0.370 → 0.389. So F1 held. Tuning k further cannot close this gap.
- **EN/FR gap is structural, not noise.** EN/FR gap narrowed but persists (EN F1 0.541 vs FR F1 0.424). The gap is more credible 
  because the sample size increased from 3 to 5 examples per language. FR retrieval remains the structural weakness. 
  This supports the case for for swapping to a multilingual embedding model (deferred).
- **Issue 2 (anglicization) did not regress.** This is consistent with (though not proof of) the defensive prompt rule holding.
- **Issue 4 (language_content_compliance volatility) did not regress.** This is consistent with (though not proof of) the defensive prompt rule holding and the increased sample size helping.
- **New citation-consistency failure mode: vague aggregate citation.** The example `science_moral_progress_en` emitted the following summary citation: 
  `[source: Lettres Philosophiques 1734, various pages]`. The logic I added to break down a multi-line citation (Issue 3) does not cover this type of 
  vague reference because there is nothing to parse from "various pages". The score for this example dropped to 0.80 on `citation_to_retrieval_consistency` due to this issue.

Conclusion:
Expanding the golden dataset validated the prior k-tuning work without producing new aggregate gains. The k=9 results in earlier sections
hold up on a larger, more demanding sample. And the EN/FR gap is now further confirmed as structural, less likely to be caused by small-sample noise.
Issues 2 and 4 did not regress, but neither did they gain new evidence of resolution beyond what the prompt rule and expanded sample provide defensively.
Further improvement on `retrieval_relevance` requires the deferred architectural changes (multilingual embeddings, smaller chunks, or a re-ranker) 
rather than further parameter tuning. The new vague-aggregate citation failure mode is logged as Issue 5 and deferred until it recurs because it is thus far isolated.

## Changes Rejected or Deferred
Listing of changes to be deferred (e.g., known gaps, missing cases), including rationales.

**Issue 1: Retrieval relevance**

- Swap embedding model. nomic-embed-text is English-leaning, which reflects the gap in the english to french scores. 
  bge-m3 or paraphrase-multilingual-mpnet-base-v2 might perform better in french while maintaining high performance in english. 
  This would be a more invasive change than tuning number of retrieved chunks and would also require re-ingestion.

- Decrease `DEFAULT_CHUNK_SIZE` from 1200 to between 600 and 800 (smaller chunks)
  Rationales: In the current state, chunks bundle multiple themes. Smaller chunks force tighter topical separation. 
  Pros: directly addresses chunk relevance without compromise
  Cons: requires regeneration of the golden dataset and re-ingestion

- Add a re-ranker or hybrid (BM25 + vector). This would likely provide the largest lift in performance, 
  but it would have large architectural impact so I'm deferring this change until after other options are attempted.

**Issue 2: Citation retrieval language**

- Parse source titles to match only on attributes like year and page, which are the same in both english and french.
  This might not work for new sources that do not include these attributes. 
  And this circumvents the user-facing problem of translating a proper noun inappropriately.

- Provide mapping of source titles to their translations in other languages. This circumvents the user-facing
  problem of translating a proper noun inappropriately.

**Issue 3: Multipage citations**

- Filter the summarized citations from the inline citations and run the evaluation only against the inline citations. 
  Inline citations do not cover multiple sources so this supports the goal of the metric while reinforcing inline citation correctness.
  However, parsing away the summarized citations is not straightforward. 

**Issue 4: Language content volatility**

- Add more golden examples to reduce volatility. Some new examples will be created as more metrics are added to the 
  eval harness and as more sources are added to the corpus. I will reconsider this approach if the volatility on 
  `language_content_compliance` persists after all metrics and data sources are added.

**Issue 5: Vague summary citation**

- Prompt the model to avoid vague aggregates like `"various pages"`. Only occurred in one experiment. Deferring until re-occurs.
- Update the consistency checker to treat unparseable summary citations as ignored rather than hallucinated. Only occurred in one experiment. Deferring until re-occurs.
