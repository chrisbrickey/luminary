## Issue Analysis
Description of top failure modes (what failed and why), including representative examples. 

**Issue 1: Retrieval relevance**

Failing to retrieve chunks that the golden dataset identifies as the most relevant is the primary failure mode (average score 0.38, well below 0.80 threshold). 

On average, fewer than 2 of the 5 retrieved chunks match expectations, with the majority being completely irrelevant. 
The problem is more pronounced with french queries (retrieval_relevance average in french 0.281 vs 0.481 in english).

The irrelevant chunks are thematically adjacent passages from the same source. 
An example are chunks about Locke's epistemology or the Quaker etiquette surfacing for "religious tolerance" 
instead of the specific passages that the golden dataset expects. 
This suggests that the embeddings capture broad topic proximity but lack the precision to distinguish among closely related passages.  


**Issue 2: Citation retrieval language**

The LLM produces citations in a different language that what the metric expects, 
which causes the `citation_to_retrieval_consistency` metric to fail (false positive).                                                                                                                      
   
In `science_moral_progress_en`, the model generated citations as "Letters Philosophiques 1734" (anglicized) 
instead of the correct French title "Lettres Philosophiques 1734" (original french). 
The consistency checker uses exact string matching against retrieved source titles. 
So all three inline citations scored as hallucinated, yielding a consistency score of 0.0 for that example. 

The issue is only present in english examples. All three french examples have consistency = 1.0 (no false positives) 
because the model renders the citations in the original french when it is prompted in French.

This is a translation issue, not a retrieval issue. The LLM translated the source titles to match the user's language.

**Issue 3: Multipage citations**
  
Aggregated multi-page citations are not matched by the consistency checker, which causes the metric to fail (false positive).                                                                                                                                             
   
In `religious_tolerance_en`, the model appended a summary citation at the end of the response: `[source: Lettres Philosophiques 1734, pages 1, 4, and 5]`. 
The checker does not recognize a match between this multi-page format and any of the individually, retrieved source title (single-page strings like `Lettres Philosophiques 1734, page 4`).
So this reduces the score for this metric even though all three pages that the model returned are correct.

**Issue 4: Language content volatility**

This issue was discovered when running multiple eval runs with different value of `DEFAULT_K`.

The metric `language_content_compliance` bounced non-monotonically across the four eval runs (1.0 → 0.833 → 1.0 → 0.833), 
as number of chunks retrieved per query (k) was tuned. But retrieval chunk count should not influence the generation-side 
language of the response. With only three English examples in the golden dataset, a single response drifting into French swings the metric by 0.33.
This metric could be stabilized by adding more English examples to the golden dataset.


## Changes Made
Listing of changes made (e.g., code, prompts, configs, datasets), including rationales.

### Retrieval Relevance

Increase `DEFAULT_K` from 5 to between 8 and 10 (more chunks per retrieval); k=9 seems to be the best choice after experimentation
- Rationales: Retrieving more chunks per query increases the likelihood that the most relevant chunks will be retrieved.
- Pros: low effort, targeted change
- Cons: trades a bit of precision for improved recall (F1 score

Experiment progression: Tuned default chunks per query (`DEFAULT_K`) 5 → 8 → 10 → 9

|                   | K=5   | K=8   | K=10  | K=9   |
|-------------------|-------|-------|-------|-------|
| Overall pass rate | 0.733 | 0.767 | 0.800 | 0.767 |
| Overall average   | 0.835 | 0.856 | 0.891 | 0.864 |
| Retrieval F1      | 0.381 | 0.447 | 0.456 | 0.489 |
| Avg recall        | 0.400 | 0.608 | 0.725 | 0.725 |
| Avg precision     | 0.367 | 0.354 | 0.333 | 0.370 |

Conclusion: K=9 is the sweet spot. Beyond K=9, additional chunks add irrelevance faster than they recover missing context.
- The K=5 → K=8 step produced the biggest gain by recovering recall (0.400 → 0.608) at almost no precision cost. 
- K=10 nudged recall higher (0.725) but precision continued to erode (0.333) and F1 nearly plateaued (+0.009 over K=8).
- At K=9, recall is already saturated (identical to K=10) while one fewer irrelevant chunk per query lifts precision back to 0.370 and yields the highest retrieval F1 in the experiment (0.489). 

Relevant eval runs:
- `2026-04-30T18-33-59.json` (k=5)
- `2026-05-27T20-02-17.json` (k=8)
- `2026-05-27T20-37-24.json` (k=10)
- `2026-05-27T20-44-18.json` (k=9)

## Changes Planned

**Issue 2: Citation retrieval language** 

- Add rule to persona prompt such as `Source titles in citations MUST be reproduced verbatim from the passages, in their original language. Never translate, anglicize, or alter source titles.`
  Rationales: The model is translating "Lettres" to "Letters" when answering in English according to the localization functionality. 
    That's not ideal for users because source titles are proper nouns that should be displayed verbatim. The metric correctly surfaced this problem.
  Pros: low effort, targeted change
  Cons: updating the prompt does not guarantee that the LLM will adhere to the rule consistently


**Issue 3: Multipage citations**

- Expand a multi-page citation into its single-page equivalents before matching. Then run the existing substring matcher.
  Rationales: For now, the presence of a citation summary is a user feature that I do not want to remove just to make the eval run easier.
    In order to ensure the summarized multi-page citations are correct, we should evaluate them as individual requirements.
  Pros: Covers both inline and summary citations (does not ignore the summary citations when running eval)
  Cons: Doesn't explicitly distinguish between inline citations and summarized citations.


## Changes Rejected or Deferred
Listing of changes to be deferred (e.g., known gaps, missing cases), including rationales.

**Issue 1: Retrieval relevance**

- Swap embedding model. nomic-embed-text is English-leaning, which reflects the gap in the english to french scores. 
  bge-m3 or paraphrase-multilingual-mpnet-base-v2 might perform better in french while maintaining high performance in english. 
  This would be a more invasive change than tuning number of retrieved chunks and would also require re-ingestion.

- Decrease `DEFAULT_CHUNK_SIZE` from 1200 to between 600 and 800 (smaller chunks)
  Rationales: In the current state, chunks bundle multiple themes. Smaller chunks force tighter topical separation. 
  Pros: directly addresses chunk relevance without compromise
  Cons: requires regeneration of the golden dataset and re-ingestion

- Add a re-ranker or hybrid (BM25 + vector). This would likely provide the largest lift in performance, 
  but it would have large architectural impact so I'm deferring this change until after other options are attempted.

**Issue 2: Citation retrieval language**

- Parse source titles to match only on attributes like year and page, which are the same in both english and french.
  This might not work for new sources that do not include these attributes. 
  And this circumvents the user-facing problem of translating a proper noun inappropriately.

- Provide mapping of source titles to their translations in other languages. This circumvents the user-facing
  problem of translating a proper noun inappropriately.

**Issue 3: Multipage citations**

- Filter the summarized citations from the inline citations and run the evaluation only against the inline citations. 
  Inline citations do not cover multiple sources so this supports the goal of the metric while reinforcing inline citation correctness.
  However, parsing away the summarized citations is not straightforward. 

**Issue 4: Language content volatility**

- Add more golden examples to reduce volatility. Some new examples will be created as more metrics are added to the 
  eval harness and as more sources are added to the corpus. I will reconsider this approach if the volatility on 
  `language_content_compliance` persists after all metrics and data sources are added.
