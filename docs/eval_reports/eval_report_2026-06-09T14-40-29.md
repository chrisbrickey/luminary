## Issue Analysis
Description of top failure modes (what failed and why), including representative examples.

### Issue 1: Keyword Coverage (new dominant gap)

Addition of the new `keyword_coverage` metric exposed a gap in the text generation functionality that previous metrics did not detect. 
Average score is 0.47 (well below the 0.60 threshold) and 6 of 10 examples failed. The EN/FR split (EN 0.60 vs FR 0.34) mirrors the gap in retrieval metrics between english and french examples.
The signal is real, but the metric is unforgiving because it expects exact stem-matching (`\b<keyword>\w{0,4}\b`) and cannot give credit for synonyms or alternative forms.

Two patterns drive the failures:
a) Lexical mismatch (responses use synonyms instead of expected vocabulary)
   The model often answers the question correctly but uses synonyms instead of words that include the expected list of word stems. 
   e.g., In `science_moral_progress_en` (score 0.4), the response uses "knowledge", "moral compass", and "scientific advancements" while the dataset expects "discoveries", "virtue", and "enlightenment".

b) Short responses (FR responses are shorter than EN responses)
   Responses in French (to FR examples) tend to be tighter than the English responses (to EN examples).
   This results in less opportunity for the correct keywords to be included. 
   e.g., In `progres_scientifique_moral_fr` (score 0.0), the response is one short paragraph that does not contain any of the targets (`philosophie`, `raison`, `découvertes`, `vérité`, `lumière`).

### Issue 2: Retrieval Relevance
K-tuning (done in previous eval round) is still saturated. The previous report deferred some architectural options 
for further addressing retrieval relevance beyond that K-tuning and those options are listed again in this report.

#### Initial investigation

Best case: `newton_fr` reaches recall 1.0 but precision is still 0.44. So F1 caps at 0.62.

The EN/FR gap on this metric persists with the larger golden example set (5 EN, 5 FR): EN F1 0.54 vs FR F1 0.42.

Thematic-adjacency (identified in previous report) is still driving this failure mode. 
Locke/epistemology and Quaker etiquette passages keep surfacing for queries about tolerance, fanaticism, and torture.

At k=9 the system is now recall-friendly but precision-bound, which matches the conclusion from the prior k-tuning experiment.

Findings based on the above table:

#### Further root cause analysis
Direct comparison of the two run artifacts confirms the retrieved chunk sets changed between the baseline (30 May 2026) and first regression run (16 June 2026). 
For example, `newton_en` went from 4 expected chunks found to 1; `newton_fr` went from 4 to 2. A chunk ID (`7ec51142de0b`) that never appears in the baseline results surfaces as irrelevant across multiple regression runs.

The following were evaluated and ruled out as causes:

| Factor | Evidence                                                                                            |
|---|-----------------------------------------------------------------------------------------------------|
| Production code | Diff `b70dee7..3243884` adds only two doc files. No src/ or scripts/ changes.                       |
| Ollama models | Both `nomic-embed-text` and `mistral` last modified ~3 months ago (before both runs).               |
| Dependencies | No commits to `uv.lock` or `pyproject.toml` between 30 May and 16 June.                             |
| Source data | All 24 raw page files last modified 25 May (before the 30 May baseline)                             |
| Golden dataset | `persona_voltaire_v3.0_2026-05-30.json` last modified 30 May 11:54 (before the baseline run at 14:39) |
| System snapshot | K=9, chunk_size=1200, same models identical in both run artifacts                                   |
| Expected chunks missing from DB | All 35 expected chunk IDs are present in the current ChromaDB collection                            |

Leading hypothesis: 
Re-ingestion (which is not git tracked) occurred between 30 May and 16 Jun. 
HNSW is an approximate nearest-neighbor algorithm whose graph structure depends on insertion order. 
Even with deterministic embeddings and the same source data, a re-ingestion with a different file iteration order would produce a new graph and marginally different top-k results for borderline queries. 
The HNSW index files show an `Apr 10` mtime (unexplained, possibly macOS APFS copy-on-write behavior), 
but the high starting seq_id of the current embeddings (3106–3312) suggests the collection has been cleared and re-ingested multiple times.

Next steps:
- Re-run ingestion from scratch.
- Then re-run the eval against v3.0. 
- If the score recovers toward 0.482, a re-ingestion fixes the regression and should be added to the standard workflow before each eval run. 
- If the score stays at 0.369, the cause lies elsewhere and a deeper investigation is needed.

#### Shift in ingested corpus is a contributing root cause
I re-ingested the corpus from scratch and ran eval against both golden datasets (v3.0 and v3.1):

Findings:
- The score did not recover to the previously seen 0.48. Both runs returned 0.371, which is statistically identical to the regression score of the previous 5 runs (0.369). 
- So the previous hypothesis (that a re-ingestion with a different HNSW insertion order caused the regression) was disproved. 
- The `data/` directory is entirely gitignored, so the raw Wikisource page files are untracked. 
  - The page files were last modified on 25 May (a re-fetch from Wikisource), which is after the April ingestion that produced the 0.482 baseline but before the May 30 baseline eval run. 
  - The April ingestion used older page content.
  - Every ingestion since May 25 uses the re-fetched content. 
  - Chunk IDs are unchanged (they hash `document_id:chunk_index`, not text), but the embedding vectors are a function of the text. 
  - If Wikisource content changed between the April and May 25 fetches, the embedding vectors for the same chunk IDs would shift - changing their cosine distance to query vectors and dropping them out of the top-k results. 
  - The 0.482 score is tied to page content that no longer exists on disk and cannot be reproduced.

Conclusions:
- 0.371 is the new true baseline. Future improvements should be measured from this number.
- The expected chunks in the golden dataset may be partially mis-calibrated against the current embedding space. They were curated when the April page content was in place. The chunks still exist by ID, but their vectors have shifted relative to the query vectors, making some harder to retrieve.
- Updating the expected chunks is not a substitute for pinning the source data. Re-curating against current text would give a valid metric baseline, but if pages are re-fetched again the same silent regression would recur.
- To address this issue appropriately, either raw source pages or checksum manifests should be committed for traceability and root cause analysis.


### Issue 3: French Response Brevity
French responses are noticeably shorter than English responses for the same topic and that appears to contribute to both Issue 1 and 2.
For example, `science_moral_progress_en` results in two full paragraphs invoking Bacon and a closing reflection while `progres_scientifique_moral_fr` results in one short paragraph. 

Shorter responses
- starve `keyword_coverage`: There is less room for expected vocabulary to appear.
- correlate with retrieval gaps: The prompt context used by the model is narrower, suggesting the retriever surfaced fewer on-topic chunks when constructing a shorter response.

When comparing French and English examples across multiple metrics, there is a strong correlation between length of response and score.  
The addition of the `keyword_coverage` metric highlighted this issue more clearly than previous metrics.

**Diagnostic to determine root cause of FR brevity:**
I analysed the eval run artifact (`evals/runs/2026-05-30T14-39-24.json`) to determine whether French response brevity is due to 
lower retrieval (pulling less chunks for French responses compared to English responses) 
or poorer performance from prompt/LLM (not responding as well in French regardless of chunks pulled). 
  
Conclusion: Chunk retrieval is a major contributing cause, especially in the worst cases. 
This means that addressing chunk retrieval for french responses (e.g. focus on multilingual embeddings) will likely 
have have a higher impact on performance than trying to control length through prompts. I could do both, but it's not adviseable
to control length via the prompt without first addressing chunk retrieval because prompting a model to be more verbose 
when context is thin produces padding - not higher quality responses.

Findings:
- The two shortest FR responses (`progres_scientifique_moral_fr` at 493 chars and `torture_justice_fr` at 647 chars) 
  both received only 2 relevant chunks (vs 4 and 3 for their EN counterparts). 
- When relevant chunk counts are equal (newton, fanatisme), FR responses match EN in length. 
- The one counter-example (`tolerance_fr`) is actually longer than its EN pair despite having fewer relevant chunks, 
  which shows the model can produce long FR output when the retrieved material is sufficiently usable.

### Issue 1: Keyword Coverage

**1B: Soften the metric (low effort, metric-side)**
Two options:
  a) Allow a small curated synonym list per keyword in the golden dataset (e.g., `"discoveries"` matches `"discovery"`, `"findings"`).
  b) Switch to embedding-based similarity (sentence-transformers cosine ≥ 0.7) so paraphrasing gets partial credit.
- Pros: reduces false negatives without changing model behavior.
- Cons: synonym lists are a maintenance burden; 
        embedding-based scoring removes the deterministic, explainable nature of the metric and adds an external dependency to the eval

Soften the metric (low effort, your choice on dataset impact)
  - Pick one sub-option:
    - Synonym list per keyword → minor dataset edit, stays deterministic.
      - **Requires:**
        - Golden dataset update: **YES** — add a synonym/variant list alongside each keyword stem for every example (extend the `expected_keywords` schema).
        - Reingestion: **NO**.
        - Re-embedding: **NO**.
    - Embedding similarity → code change only, no dataset edit, but loses determinism and adds an eval-side dependency.
      - **Requires:**
        - Golden dataset update: **NO** — existing keyword list is reused as the reference text for cosine comparison.
        - Reingestion: **NO**.
        - Re-embedding: **NO** of the corpus; the eval harness gains a runtime dependency on a sentence-transformers model invoked at scoring time.
  - I'd lean synonyms — the determinism is worth more than the maintenance cost at this stage, and it composes with the Phase 1 audit (same file edit).

- Pros: simple code change, directly addresses retrieval relevancy

**1C: Reduce noise by making the LLM responses more deterministic (generation-side)**

Added a default `temperature=0` and a fixed default `seed` to the chat-chain configuration to reduce LLM responses non-determinism. 
Both are production defaults, applied at the boundary where the chat model is constructed. No changes to ingestion, embedding, metrics, or golden dataset.

Why both temperature and seed:
  - temperature=0 forces Ollama to use greedy decoding (no sampling). This is the dominant determinism lever.
  - seed covers tie-breaking decisions and any path that still uses random number generation in Ollama

Experiments:
After implementing the change, I ran the eval harness four times against the most recent dataset (`persona_voltaire_v3.1_2026-06-16`). 
Runs A and C were the first eval after an Ollama restart (cold). Runs B and D were the second eval within the same Ollama session (warm). 
All other system inputs (models, K, chunk size, dataset, code) were held constant.

| Run | Artifact                              | Ollama state                         |
|---|---------------------------------------|--------------------------------------|
| A | `evals/runs/2026-06-21T12-43-53.json` | fresh start                          |
| B | `evals/runs/2026-06-21T12-45-38.json` | warm (2nd in session, after A)       |
| C | `evals/runs/2026-06-21T12-56-25.json` | fresh start (post-restart of Ollama) |
| D | `evals/runs/2026-06-21T13-00-46.json` | warm (2nd in session, after C)       |

Findings:
- Aggregate scores were identical across all four runs: `overall_pass_rate=0.7333`, `overall_average=0.8102`, including every entry for `averages_by_metric` and `averages_by_language_and_metric`.
- All 60 per-example metric scores (6 metrics × 10 examples) are identical across all four runs. All 10 pass/fail outcomes match.
- Raw response text fell into two reproducible clusters keyed by Ollama warm/cold state: cold-start runs (A ≡ C) are byte-identical to each other, and warm-session runs (B ≡ D) are byte-identical to each other. The two clusters differ on the same four examples every time: `tolerance_en`, `newton_en`, `judicial_torture_en`, `torture_justice_fr`.
- Differences between the two clusters are at the wording level only and did not change any metric score in any cell. So any residual non-determinism does not propagate to the eval signal.
- The two-cluster pattern is consistent with Ollama selecting different kernel/batching strategies after first-run warmup, not with stochastic LLM noise. 
- With temperature=0 and a fixed seed, the sampling path is removed. What remains is a deterministic but state-dependent execution path inside the serving layer.

Conclusions:
1. The stated objective is achieved. Eval reports are reproducible at the metric and aggregate level. Deltas can now be attributed to system changes rather than LLM jitter.
2. No further determinism work is justified at this stage. The remaining non-determinism is bounded, does not move metrics, and the marginal cost of chasing it (e.g., pinning Ollama batching or moving off the local server) far exceeds the value over an eval workflow that already produces stable scores.
3. Text-level reproducibility is achievable but conditional: restart Ollama before each baseline run to land reliably in the cold-start cluster. This is a cheap operational recipe to add to the eval workflow when exact text diff matters.
4. The four examples that drift between warm/cold clusters (`tolerance_en`, `newton_en`, `judicial_torture_en`, `torture_justice_fr`) are a useful watchlist. They share the property that small numerical perturbations compound into wording differences.


### Issue 2: Retrieval Relevance

**2A/2C: Swap to a multilingual embedding model and regenerate expected chunks in golden dataset (medium effort)**
In an attempt to improve retrieval from French responses (and future retrieval in additional languages), 
I changed the default embedding model (for vector store operations) from `nomic-embed-text` (768 dimensions) to a multilingual model: `bge-m3` (1024 dimensions). 
I re-embedded the corpus to the vector store and updated the expected chunks retrieved in the golden dataset (bumped from `v3.1` to `v3.2`) using LLM as judge. 
This update to the golden dataset addressed both the change to the embedding model and the unexpected shift in the ingested corpus that occurred prior to the start of this eval report.

_NB: I also considered `paraphrase-multilingual-mpnet-base-v2` for the embedding model. I rejected it in favor of `bge-m3` because it would have required additional changes to the codebase. `bge-m3` works automatically with the current ollama provider._

- Pros: Highest-confidence improvement for French responses across multiple metrics because this change addresses chunk retrieval directly, 
  which is also a contributing factor to lower keyword coverage and brevity of French responses.
- Cons: Invasive change and baseline behavior on English reponses may shift. May require retuning of DEFAULT_K (number of chunks retrieved per query).

Experiments:
I ran the eval harness four more times with the chat model, K, and chunk size held constant. I also restarted Ollama before every run ('cold state').
The first two runs were against the old v3.1 dataset. The last two runs were against the new v3.2 dataset.
Both bge-m3 + v3.1 runs and both bge-m3 + v3.2 runs were byte-identical at the aggregate and per-example level, confirming determinism, so they are displayed as single lines below.

| Regime | Artifact | Embedding | Dataset | rr (EN) | rr (FR) | rr (all) | kc (all) | pass rate | avg |
|---|---|---|---|---:|---:|---:|---:|---:|---:|
| Pre-swap baseline | `evals/runs/2026-06-19T17-23-12.json` | nomic-embed-text | v3.1 | 0.349 | 0.393 | 0.371 | 0.473 | 0.717 | 0.791 |
| Post-swap, stale dataset | `evals/runs/2026-06-21T14-32-26.json` | bge-m3 | v3.1 | 0.217 | 0.282 | 0.250 | 0.367 | 0.717 | 0.770 |
| Post-swap, refreshed dataset | `evals/runs/2026-06-21T15-11-23.json` | bge-m3 | v3.2 | 0.551 | 0.462 | 0.506 | 0.367 | 0.733 | 0.812 |

Findings:
- `retrieval_relevance` rose +0.135 absolute (+36% relative) over the pre-swap baseline once the golden dataset was recalibrated to the new embedding space.
- The intermediate dip on the stale v3.1 dataset (0.371 → 0.250) was a measurement artifact, not a model regression. The metric scores overlap with `expected_chunks`, and `expected_chunks` in v3.1 was curated against the nomic embedding space. Refreshing the dataset (v3.2) swung the same retrieved chunks from 0.250 → 0.506.
- Per-example wins from bge-m3 + v3.2 are concentrated where nomic was weakest: `science_moral_progress_en` 0.400 → 0.800, `fanaticism_en` 0.400 → 0.750, `progres_scientifique_moral_fr` 0.143 → 0.571, `newton_fr` 0.462 → 0.625.
- Multilingual embeddings lifted both languages rather than preferentially lifting FR. The EN/FR retrieval gap did not close in the originally-stated direction. Both languages improved in absolute terms, but EN improved more (EN 0.349 → 0.551, +0.202; FR 0.393 → 0.462, +0.069). So EN now leads FR by 0.089.
- `keyword_coverage` is unchanged between the v3.1 and v3.2 bge-m3 runs (0.367 in both). v3.2 only regenerated `expected_chunks`, not `expected_keywords`, so the keyword stems are still the v3.1 set. The metric is now uncoupled from retrieval quality. Addressing it requires the metric-side change, which I will do subsequently.
- Other metrics held at 1.000 across all runs (`citation_accuracy`, `citation_to_retrieval_consistency`, `language_metadata_compliance`, `language_content_compliance`). The swap did not destabilize the generation pipeline.

Conclusions:
1. Adopting `bge-m3` as the new default embedding model delivered a +0.135 lift on `retrieval_relevance` and +1.6pp on overall pass rate against a calibrated dataset.
2. The original motivation (closing the EN/FR retrieval gap in favor of FR) was not achieved. Both languages benefited but EN benefited more. This is worth noting before assuming further multilingual changes will help FR specifically.
3. `retrieval_relevance` is still below the 0.80 pass threshold in both languages (EN 0.551, FR 0.462). K retuning is the next lever and is now well-positioned: the v3.2 dataset provides a stable ground truth in the bge-m3 space.
4. `keyword_coverage` is now the dominant failure mode and is no longer correlated with retrieval. So after retuning K, metric softening (another change planned) should be implemented.


### Issue 3: French Response Brevity

**3A: Limit length of LLM responses (low effort, generation-side)**

The diagnostic to determine root cause of brevity revealed that it is driven heavily by chunk retrieval: fewer chunks
are retrieved for French queries resulting in shorter responses (less content). English responses score higher across many
metrics because they cover more ground, but they are generally longer than is ideal for a conversational app. 
Responding with a wall of paragraphs does not encourage debate. Encouraging English responses to be more focused will 
improve the user experience and lessen the gap in average response length between the two languages. So the following
was added to Voltaire's persona prompt.

`**Concision**: Limit your ENTIRE response to ONE short paragraph of AT MOST 4 sentences, regardless of how many topics the question raises. Address the most important angle directly.`

- Pros: Low effort; Can be A/B tested in a single run
- Cons: Smaller surface area of response will likely result in an initial performance decline on english reponses (less opportunity to check a box)
- Rationale: Improvement to user experience and a long-term 'leveling' of the surface area available for both English and French 
  responses (one less variable to consider) is worth the short-term decline in performance of english responses. 

Experiments:
Ran two A/B comparisons against the pre-change baseline (the old version of `voltaire.py`). Only the prompt's sentence cap varied.
All 3 runs share the same models (`mistral`, `nomic-embed-text`), golden dataset (`persona_voltaire_v3.1_2026-06-16`), K (9), and chunk size (1200).

| Run | Prompt cap | Artifact |
|---|---|---|
| Baseline | no hard cap (3–5 sentences per topic) | `evals/runs/2026-06-19T17-23-12.json` |
| Experiment A | AT MOST 4 sentences | `evals/runs/2026-06-20T09-34-33.json` |
| Experiment B | AT MOST 5 sentences | `evals/runs/2026-06-20T09-50-39.json` |

Aggregate metrics (citation_accuracy and language_metadata_compliance held at 1.000 across all three runs):

| Metric | Baseline | 4-sent | 5-sent |
|---|---:|---:|---:|
| Overall pass rate | 0.717 | 0.700 | 0.700 |
| Overall average | 0.791 | 0.805 | 0.800 |
| citation_to_retrieval_consistency | 1.000 | 0.967 | 1.000 |
| keyword_coverage (EN / FR) | 0.48 / 0.47 | 0.40 / 0.58 | 0.44 / 0.42 |
| language_content_compliance | 0.900 | 1.000 | 1.000 |
| retrieval_relevance | 0.371 | 0.371 | 0.371 |

Response length (not an official metric):

| | Baseline |  4-sentence cap |       5-sentence cap |
|---|---:|----------------:|---------------------:|
| EN avg sentences / words | 6.4 / 172 |       4.0 / 117 |            5.0 / 121 |
| FR avg sentences / words | 6.0 / 162 |       5.4 / 130 |            4.0 / 100 |
| EN/FR word gap | +10 (EN longer) | −13 (FR longer) | +22 (EN much longer) |

Findings:
- 4-sentence cap (Experiment A) achieved the stated goal. 
  - EN examples averaged exactly 4 sentences (cap saturated). 
  - The EN/FR length gap closed and slightly inverted (+10 → −13). `language_content_compliance` lifted 0.90 → 1.00. 
  - `keyword_coverage` FR gained +0.11 (likely thanks to increased surface area) while EN lost −0.08 (the model shed peripheral terms to fit the sentence cap). 
  - Net overall average ticked up despite the EN keyword tradeoff.
- 4-sentence cap introduced one citation hallucination. `science_moral_progress_en` invented "Lettres Philosophiques 1734, page 29" 
  (not in the retrieved set), pulling `citation_to_retrieval_consistency` from 1.000 to 0.967. This might have been due 
  to the model being pressured to compress justification of the content into a smaller surface.
- 5-sentence cap (Experiment B) cleared the citation hallucination (back to 1.000) but the FR scores regressed sharply. 
  - Every FR example shortened. FR avg fell below both baseline and cap=4.
  - FR keyword coverage dropped 0.58 → 0.42, potentially meaning that the model used the extra room to paraphrase rather than to add key vocabulary.
  - The EN/FR word gap re-inverted to +22, worse than the original +10.
- Retrieval relevance is identical (0.371) across all 3 runs, confirming the change only impacted generation-side behavior.
- Summary: cap=4 acts as a homogenizing pressure, forcing both languages into a tight, focused frame with the same surface area.
  cap=5 is permissive enough that the model's natural FR brevity reasserts itself and the original problem returns.

Conclusions:
1. Adopt sentence cap=4 as the new baseline for persona prompts because it is the better tradeoff. 
2. cap=4 delivers the benefit (closed length gap, +0.10 language compliance, +0.11 FR keyword) at the 
   cost of one citation hallucination. cap=5 trades that hallucination back for a re-opened (and worse) length gap, systematic 
   FR shortening, and a −0.16 FR keyword regression (net negative on the stated objective).
3. The citation hallucination is a watch-item, not a blocker. If it recurs, I could address with a prompt rule against speculative page pins rather than relaxing the cap.

## Changes Deferred
Listing of changes to be deferred (e.g., known gaps, missing cases), including rationales.

**Issue 1: Keyword Coverage**

- Tighten the persona prompt to favor canonical vocabulary (low effort, generation-side)
  Add a rule along the lines of: "When you cite an idea, name it with the term Voltaire would use
  (`tolérance`, `liberté de conscience`, `superstition`, `philosophes`, `lumières`) rather than a modern paraphrase."
  Pros: targeted, low effort, and matches what the metric expects
  Cons: nudging vocabulary risks distorting the model's natural rendering; requires validation that responses still read well

**Issue 2: Retrieval Relevance**

- Hybrid retrieval (BM25 + vector) or re-ranker (high effort)
  Biggest expected lift but most invasive change. The previous report deferred this option and I am deferring again. 
  This should be viewed as a last resort in some way because once a re-ranker is in place, 
  it becomes difficult to determine the relative impact of other changes.

- Pin the raw source pages (medium effort)
  The `data/` directory is gitignored, so the raw Wikisource pages are untracked. When a regression in performance against
  the retreival relevance metric occurs, the system cannot detect or examine changes in the ingested corpus. A silent 
  re-fetch from Wikisource can shift embedding vectors and degrade eval scores with no audit trail. I suspect this the 
  reason for the punctuated and persisting drop in retrieval relevance around May 2026, but I can't prove it.

  Two options were considered to address this:
    a) Commit the raw pages to git (data is currently only ~268 KB across 24 files) and have ingestion verify a checksum
       manifest before chunking. This both detects drift AND makes prior snapshots recoverable via `git checkout`.
    b) Commit only a checksum manifest (raw pages remain gitignored) and have ingestion verify on-disk files against it.
       This detects drift but cannot recover lost snapshots.

  Pros: Surfaces source-data drift to prevent future regressions or to aid in root cause analysis
  Cons: Commits text content into the app repo or adds a bootstrapping step (manifest-only variant) where fresh
        clones need an initial scrape before ingestion; Requires changes to `scripts/ingest.py` and `scripts/scrape_wikisource.py`
        and the addition of a `corpus_version` field to eval run artifacts.

  Rationale for deferral: This is the most rigorous fix for the regression documented above, but it is more scope than
  I want to take on right now. The 0.371 post-reingest score is adopted as the new baseline, against which future retrieval
  improvements will be measured. I recognize that if Wikisource content changes again before this is implemented, 
  the same silent regression could recur and could consume significant investigation to diagnose.

## Changes Rejected

**Issue 3: French Response Brevity**

- Add a minimum length/depth instruction to the persona prompt (low effort)
  Add a line to the persona prompts such as: "Respond with a minimum of 3 to 5 sentences. Cite at least two passages."
  This directly addresses Issue 3 and indirectly addresses Issue 1 (more text results in more keyword surface area). 
  Pros: low effort; can be A/B tested in a single run
  Cons: Forcing minimum length without improving chunk retrieval will likely induce the model to pad the responses 
        instead of producing more pertinent responses.; Readility could decline.; 
        This change would require qualitative review of the result text (not just review of resulting metrics).

- Static translated persona prompts (medium effort)
  Dynamic runtime translation of the prompt would add an LLM call per query which would be too detrimental to latency and user experience. 
  So translated prompts would need to be static (pre-translated) and selected at runtime according to the already available detected language.
  Cons: Does not address root cause. When retrieved chunk counts are equal, French responses match English responses in length. So chunk retrieval is driving brevity. Changing the prompt language is not likely to impact brevity.
        Requires a translated prompt for every formally supported language
        Increases risk of lack of comprehension by the model when detected language is a minor language
