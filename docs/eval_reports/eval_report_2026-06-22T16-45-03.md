## Issue Analysis

### Issue 1: Keyword Coverage
`keyword_coverage` is a remaining metric failure: aggregate 0.53, which is below the 0.60 threshold.
The LLM responds with roughly 50% of the expected terms overall.
French examples perform worse than English examples: FR 0.49 vs EN 0.56.

**Background:**

Prior tuning attempts (fully documented in `eval_report_2026-06-09T14-40-29.md`) worked across multiple levers on this metric, but plateaued below threshold.

| Previous changes attempted       | Details                                                                                               | Effect on `keyword_coverage` metric                                                                                                   |
|----------------------------------|-------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------|
| Dataset keyword audit            | Swapped 4 stems for Voltaire-era fairness (v3.0 → v3.1, e.g. `enlightenment` → `knowledge`)           | Aggregate Δ −0.015. This was a wash without meaningful signal.                                                                        |
| Generation determinism           | `temperature=0` + fixed `seed` on Ollama                                                              | Per-example scores became reproducible; No direct lift but unblocked further tuning.                                                  |
| Embedding swap + dataset refresh | `nomic-embed-text` (768d) → `bge-m3` (1024d, multilingual); Refreshed `expected_chunks` (v3.1 → v3.2) | Unchanged (0.367); Expected chunks were updated but expected stems were not.                                                          |
| K-tuning sweep (K=3…15)          | Adopted K=10 as new `DEFAULT_K` (best on 3 aggregate scores)                                          | Best keyword_coverage observed: 0.527 at K=10. This is still below the 0.60 threshold. The scores were non-monotonic across K values. |

**Ongoing issues:**

'High performance' for this metric should mean that the app is responding on topic and in alignment with the selected philosophe person (within reason).
But the current keyword stems and regex used by the metric (`\b<stem>\w{0,4}\b`) results in success only when the response 
begins with the single selected keyword stem and continues on for ≤4 chars (case-insensitive).

Shortcomings of the current metric design:
- It does not count words where the stem occurs in the middle of the word. 
- It does not count words that start with the stem and continue on for more than 4 characters.
- The suffix character count portion of the regex (`w{0,4}`) does not apply to multi-word keywords. Those only count as found if the text is an exact match.
- It does not count synonyms. We want the LLM to be liberated to select words as long as it adheres to the expected subject matter.

**Conclusions:**
- It's possible that the current behavior of the app might be acceptable (despite the currently failing score on this metric) 
  if the LLM is using a variation on the keyword or a synonym that is on-topic but does not succeed based on the current stem + regex rule.
- Potential solutions include:
  - tuning the regex expression
  - tuning the length of the stems (e.g. philos vs philosoph)
  - tuning the choice of keyword stems (e.g. superstition vs croyance, or a shortened version of these). 
  - update the metric to allow the use of synonym stems for each keyword stem
  - _NB: Tuning stems and regex introduce tradeoffs. The more flexible we lean (e.g. short stems and many characters allowed), 
the more likely the metric will count off-topic words as successful. This would also apply to the length of any synonym stems added._

## Changes Made

### A. Tune stem length of the expected keywords

To address below-threshold performance on the `keyword_coverage` metric, I began by tuning keyword stem length, 
resulting in an updated golden dataset (`v3.3`). Then I ran the eval harness again (artifact: `2026-06-25T14-47-50.json`).

**Rationale:**
- Tuning stem length can be done per-word, flexing to how that specific word is used in natural language. This provides much finer control compared to tuning the regex expression, which applies in a blanket-manner to all stems.
- Tuning stem length does not require a code change, compared to adding synonym lists to the metric which would require a medium-sized code change.

**Notable decisions:**
- The stems `"philosophy"` (EN) and `"philosophie"` (FR), were chosen especially to avoid an example passing due to the commonly cited source `Lettres Philosophiques 1734` (as opposed to real response content).
  We could use `philosoph` in both languages if we could guarantee that the character allowance of regex expression never changed (because the 5-char suffix `iques` exceeds the current 4-char regex suffix limit).
  I want to avoid reliance on that unenforced coupling between two keyword stems and the regex suffix limit - something ripe for future regressions.
- The one multi-word primary keyword `"liberté de conscience"` was rewritten as two single-word stems: `"libert"` and `"conscience"`. 
  Why? So that each keyword represents only one meaning and so that the regex expression can be applied to all keywords. (The current regex rule short-circuits on multi-word stems.)
  This change from one multi-word keyword to two single-word keywords is expected to have an initial negative impact on performance because it increases the number of keywords that must be found, increasing the denominator.

- The alignment keyword stems within EN/FR example pairs revealed that the model could achieve language parity (EN/FR gap of 0.0) at least within the confines of the 5 subject areas tested.

## Changes Deferred

### Add synonym list for each keyword stem in the keyword_coverage metric

_PRIORITIZED FOR SUBSEQUENT UPDATE TO THIS EVAL REPORT_ 

Expand the keyword_coverage metric to use a list of synonyms for each keyword stem.
Update the golden dataset (minor version bump) with a curated list of synonym stems for each keyword in every example.
When developing synonym stems, adhere to the principles used when determining length for the keyword stems (because the same regex expression will be applied to both keyword and synonym stems).
Update the golden dataset generation tooling, which includes at least one script and at least one markdown file.

Pros: Maintains determinism (vs using embedding-based scoring); Reduces false negatives without changing model behavior; No re-ingestion or re-embedding required

Cons: Synonym list adds a maintenance burden; Requires a medium-sized code change and an update to the golden dataset

Rationales for deferal: 
- Before developing a list of synonyms per keyword, I wanted to make sure that the keyword stems themselves were appropriate. 
  So I chose to implement the other keyword changes in this report before adding lists of synonyms.
- Adding synonym lists will significantly soften the `keyword_coverage` metric by justifiably permitting/encouraging 
  the LLM to make specific word choices as long as the response remains on-topic. It would have been more difficult to
  segregate performance impact of keyword stems vs impact of synonym stems if I had implemented the synonyms lists first.

## Changes Rejected

### Update the regex logic of the keyword_coverage metric

Pros: small code change; easy to A/B test; could address the issue where the stem occurs in the middle of a word

Cons: increases risk of irrelevant words counting as 'found'

Rationales for rejection:
- Adjusting stem length (see 'Changes Made' section) provides more fine-tuned control, compared to adjusting the regex expression, 
  because it is a per-word adjustment that can be tailored to how each keyword is used in natural language.
  A change to the regex expression would be applied globally across all keyword stems.
- The benefit of easier A/B testing with regex expressions is outweighed by the risk of accepting irrelevant words 
  for some stems in order to optimize the regex or other stems.
